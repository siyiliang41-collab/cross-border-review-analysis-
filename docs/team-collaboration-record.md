## 四、接口约定文档

### 4.1 数据采集 → 数仓

```
格式：CSV（UTF-8，逗号分隔，RFC 4180）
字段：23列（productId, buyerEval, buyerCountry, feedback, feedbackTranslated,
       evalDate, skuInfo, logistics, hasImage, hasFollowUp, addFeedback, anonymous,
       label1~3, labelValue1~3, ...）
注意：feedback/feedbackTranslated 等文本列含多行文本，入库前需扁平化处理
```

### 4.2 数仓 → 后端

```
格式：MySQL 表（bipt_project 库）
表名前缀：ads_*（应用数据服务层）
13张表：ads_product_rating, ads_country_top15, ads_monthly_trend,
        ads_logistics, ads_image_followup, ads_sku_analysis,
        ads_sentiment_product, ads_product_scorecard,
        ads_country_product_matrix, ads_feature_country_product,
        ads_sku_country_pref, ads_country_monthly_trend
```

### 4.3 后端 → 前端

```
协议：HTTP GET（RESTful）
返回格式：JSON
统一结构：{ code: 200, message: "success", data: [...] }
跨域：CORS 全局允许（开发阶段）
API 前缀：http://192.168.229.101:8080/api
```

## 五、经验总结

### 5.1 做得好的

1. **角色分工清晰** — 5人每人有明确职责，各司其职无重叠
2. **并行开发高效** — 数仓/后端/前端三线并行，2.5天完成6层全链路
3. **Bug记录详实** — 7个问题每个都有根因分析+排查过程+最终方案，答辩时这些都是加分故事
4. **技术选型务实** — Sqoop不行就换Spark，Hive MR OOM就改Spark计算，不死磕

### 5.2 可以改进的

1. **需求文档不够正式** — 当前多为口头+微信沟通，缺少书面需求规格说明书
2. **进度记录后补** — 当时的站会没有留下书面记录，本文档部分内容是基于代码提交历史回溯的
3. **缺少周报制度** — 本次开发密集只有3天，但v3.0有3周，应建立周报制度

### 5.3 v3.0 改进措施

- 每周日 22:00 周报截止，梁思怡汇总
